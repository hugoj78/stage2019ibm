Serverless

The serverless is a model in which the cloud service provider (IBM Cloud, AWS, Azure or Google Cloud) is responsible for executing a piece of code by dynamically allocating resources. And he only bills the amount of resources used to execute the code. The code is usually executed in non-standard containers that can be triggered by various events, including http requests, database events, queue services, surveillance alerts, file downloads, scheduled events (cron tasks), etc. The code sent to the cloud provider for execution is usually in the form of a function. Therefore, serverless is sometimes referred to as “Functions as a Service” or “Faas”.

The Serverless Movement

Serverless abstracts away the most menial parts of building an application, leaving developers free to actually spend their days coding.

What this means is that developers can single-handedly build apps that handle production-ready traffic. They don’t have to actively manage scaling for their applications. They don't have to provision servers, or pay for resources that go unused. They can just get projects off the ground with small, agile teams.

Thousands of developers are already proving that serverless can enable them to launch applications at record speed and cost. Adoption has been fast, with companies like Coca-cola, Nordstrom, EA, and Expedia touting their number of serverless services in production.

Ultimately, serverless is about focusing your efforts on what provides value to users. Upgrading your Linux distro does not provide value to users. Managing your RabbitMQ servers does not provide value to users. Shipping product provides value to users.

That is the serverless maxim: focus on business logic, not servers. Serverless has become a movement about faster development and creative empowerment.

What's Serverless


Zero administration

Deploy code without provisioning anything beforehand or managing anything afterward. There is no need to provision or maintain any servers. There is no software or runtime to install, maintain, or administer.

Auto-scaling

Your application can be scaled automatically or by adjusting its capacity through toggling the units of consumption rather than units of individual servers.

Pay-per-use

Function-as-a-service compute and managed services charged based on usage rather than pre-provisioned capacity. You never pay for resources you don't use.

Operate with the Serverless Dashboard

The serverless Dashboard provides architectural views of your serverless applications. It also exposes logs metrics, alarms and debugging, and lets you collaborate easily with teammates on development.

IBM Cloud Functions

Based on Apache Openwhisk, IBM Cloud Functions is a multilingual Faas (Function As A Service) programming platform that enables the development of lightweight code that is scalable upon request.

Explore an open ecosystem

IBM Cloud Functions provides access to the Apache Openwhisk ecosystem in which everyone can bring their own action code as building blocks of the expanding repository.

Accelerate and simplify development

IBM Cloud Functions accelerates application development, which enables developers to quickly build apps with action sequences that execute in response to our event-driven world.

Take advantage of the benefits of cognitive services

Pay only for what you use

It’s simple: pay for actual use rather than pre-allocated capacity. Costs increase only as you construct more OpenWhisk-intensive solutions or as existing solutions scale to support bigger loads.

Use case

As an event-driven platform, Cloud Functions supports a variety of use cases.

This includes the generation of web applications and Apis. With web applications, events are the interactions between web browsers (or REST clients) and your web application, namely HTTP requests. Instead of making a virtual machine, container or Cloud Foundry execution environment available to deploy your back-end, you can implement your back-end API with a server-free platform. This solution allows to avoid the costs of idle time and to size the platform according to the needs.

Any action (or function) in Cloud Functions can be turned into an HTTP endpoint ready for use by web clients. When enabled for the Web, these actions are called Web actions. Once you have Web actions, you can assemble them into a full API using Gateway API. API Gateway is a Cloud Functions component that exposes Apis. It offers security features, Oauth support, speed limitation, custom domain support.
